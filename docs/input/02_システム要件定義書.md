## 1. システム全体構成

### 1-1. アーキテクチャ概要

```
【参加者側】
LINE公式アカウントに「応募」送信
  → GAS Webhook（doPost）
      → 受付コード生成・reply返信
      → 会員マスタに記録
      → アクション履歴に記録

LINE公式アカウントに「応募状況」送信
  → GAS Webhook（doPost）
      → 設定シートの全イベントを走査
      → 各イベントの当落シート・応募シートでUser IDを確認
      → ステータスをまとめてreply返信
      → アクション履歴に記録

Googleフォームに入力・送信
  → GAS フォームトリガー（onFormSubmit）
      → 受付コードで会員マスタからUser IDを取得
      → 突合成功：応募シートにUser ID追記・当落シートに転記
                   個人宛push送信・スタッフグループにログ
                   会員マスタの名前・最終更新日時を更新
                   アクション履歴に記録
      → 突合失敗：スタッフグループにアラート・メール送信

【スタッフ側】
ダッシュボード（GAS HTMLサービス）
  → イベント一覧・応募者確認・当落入力・通知送信・絞り込み送信

notify-result.gs（ダッシュボードまたはメニューから呼び出し）
  → 当落シートの未送信行にLINE push送信
  → アクション履歴に記録
  → スタッフグループに送信完了ログを投稿
```

### 1-2. 技術スタック

| レイヤー | 採用技術 | 理由 |
|---|---|---|
| 実行環境 | Google Apps Script（GAS） | 追加インフラ不要・GoogleサービスとのネイティブAPI連携 |
| データ管理 | Googleスプレッドシート | スタッフが直接確認・編集可能 |
| 管理UI | GAS HTMLサービス（HtmlService） | 追加サーバー不要・GASから直接HTML配信 |
| メッセージング | LINE Messaging API | 既存LINE公式アカウントを活用 |
| アラートメール | GmailApp（GAS組み込み） | 受取アドレスをスクリプトプロパティで管理 |
| フォーム | Googleフォーム | 既存フォームを流用（受付コード欄追加のみ） |
| 認証情報管理 | GASスクリプトプロパティ | コード直書き禁止 |

## 2. GAS実装設計

### 2-1. スクリプト構成

| ファイル | 役割 |
|---|---|
| `config.gs` | シート名定数・スクリプトプロパティアクセサー |
| `utils.gs` | 共通ユーティリティ（コード生成・LINE API送信・アクション履歴記録・イベント一覧取得） |
| `webhook.gs` | LINE Webhookの受信・「応募」処理・「応募状況」照会 |
| `form-trigger.gs` | フォーム送信時の突合・通知・転記・会員マスタ更新 |
| `notify-result.gs` | 当落結果の一括LINE送信・スプレッドシートのonOpen |
| `dashboard.gs` | GAS HTMLサービス。管理ダッシュボードのUI配信とデータ操作API |
| `setup.gs` | 初回セットアップ・新イベント追加 |

### 2-2. 各スクリプトの詳細

#### config.gs

```javascript
const SHEET = {
  CONFIG: '設定シート',
  MEMBERS: '会員マスタ',
  ACTION_LOG: 'アクション履歴',
};
function getProp(key) { ... }
```

#### utils.gs

```
generateCode()
  → 紛らわしい文字（O・0・I・1）を除いた英数字8文字

getSheet(name)
  → SPREADSHEET_IDのスプレッドシートから指定シートを取得

getAllEvents()
  → 設定シートの全行（2行目以降）を走査して返す
  → 各要素: { name, eventDate, closingDate, appSheetName, resultSheetName, winMsg, loseMsg }

logAction(userId, actionType, eventId, detail)
  → アクション履歴シートに1行追加
  → 列: タイムスタンプ・User ID・アクション種別・イベント識別名・詳細

linePost(endpoint, payload)
replyMessage(replyToken, text)
pushMessage(to, text)
notifyStaff(text)
sendAlertEmail(subject, body)
```

#### webhook.gs

```
doPost(e):
  → bodyをパース → messageイベントを走査
  → text.trim()で完全一致判定

handleOubo(event):
  1. 会員マスタB列でUser IDを検索
  2. 既存あり → 同行C列の受付コードをreply再送・アクション履歴「既存コード再送」
  3. 新規 → generateCode() → 会員マスタに [登録日時, userId, code, '', '', ''] を追加
             → replyで受付コードを返信・アクション履歴「受付コード発行」

handleOuboStatus(event):
  1. getAllEvents() で全イベント取得
  2. 各イベントについて開催日 < 今日 ならスキップ
  3. {識別名}_当落シートB列でUser IDを検索 → 当選/落選
  4. なければ {識別名}_応募シートS列（インデックス18）でUser IDを検索 → 応募済み（当落発表前）
  5. なければ募集終了日で期間中/終了を判定
  6. 全イベント分をまとめて1通でreply
  7. アクション履歴「応募状況照会」
```

#### form-trigger.gs

```
onFormSubmit(e):
  ※スプレッドシートの「フォーム送信時」トリガーとして設定（初回1回のみ）
  ※複数イベントのフォームを同一スプレッドシートに連携すれば1つのトリガーで全対応

  1. e.namedValues['受付コード'][0] を取得・trim・toUpperCase
  2. e.namedValues['お名前'][0] を取得
  3. e.range.getSheet() で応募シートを取得・シート名を確認
     → {識別名}_当落シート名を導出（_応募 → _当落）
  4. 重複チェック：今回行以外で同じ受付コード＋T列「済」があればアラートして終了
  5. 会員マスタC列を走査（大文字比較）でUser IDを取得
  6. User IDなし → スタッフLINEグループにアラート＋GmailAppでメール送信・アクション履歴「突合失敗」→ 終了
  7. User IDあり:
     a. 応募シートS列（19列目）にUser IDを記録
     b. 応募シートT列（20列目）に「済」を記録
     c. {識別名}_当落シートにお名前・User IDを転記（C〜E列は空欄）
     d. push APIで参加者個人宛に「応募完了」通知
     e. スタッフLINEグループに受付完了ログ
     f. 会員マスタのUser ID行のE列（名前）・F列（最終更新日時）を更新
     g. アクション履歴「応募完了」

列インデックス（0始まり）:
  B列 = index 1（受付コード）
  C列 = index 2（お名前）
  S列 = index 18（User ID・GAS追記）
  T列 = index 19（済・GAS追記）
列番号（1始まり・getRange用）:
  S列 = 19、T列 = 20
```

#### notify-result.gs

```
sendResults():
  1. getActiveSheet() で当落シートを取得
  2. シート名が「_当落」で終わることを確認
  3. getResultMessages(sheetName) で設定シートのE列・F列からメッセージ文を取得
  4. 2行目から走査。D列「済」はスキップ・C列が「当選」「落選」以外もスキップ
  5. push APIで個人宛に当落通知
  6. D列に「済」・E列に送信日時を記録
  7. アクション履歴「当落通知_当選」または「当落通知_落選」
  8. 10件ごとに Utilities.sleep(1000) で1秒待機（LINE APIのレート制限対策）
  9. 全完了後スタッフグループに送信ログ（当選N名/落選N名）

getResultMessages(resultSheetName):
  → 設定シートD列を参照して対応するイベント行のE列（当選文）・F列（落選文）を取得
  → 空欄の場合はデフォルト文を返す

onOpen():
  → 「イベント管理」メニューに「当落通知を送信」「新しいイベントをセットアップ」を追加
```

#### dashboard.gs

```
doGet(e):
  → GAS HTMLサービスでダッシュボードHTMLを返す
  → アクセス権：GASデプロイ設定で「自分のみ」または「全員」を選択

【クライアントサイドから呼び出すサーバー関数】

getEventsData():
  → 設定シートの全イベントを返す
  → 各イベントに応募者数・当落済み数・未処理数を付加

getApplicants(appSheetName, resultSheetName):
  → {識別名}_応募シートと{識別名}_当落シートを結合
  → 応募者ごとに: { name, userId, appliedAt, result, sent } を返す

setResult(resultSheetName, userId, result):
  → {識別名}_当落シートのUser IDに一致する行のC列に当選/落選を書き込む
  → resultは「当選」または「落選」のみ受け付ける

sendResultsFromDashboard(resultSheetName):
  → notify-result.gsのロジックをダッシュボードから呼び出す

getFilteredUsers(appSheetName, resultSheetName, status):
  → status: '未応募' | '応募済み' | '当選' | '落選'
  → 対象User IDの一覧を返す

sendBroadcast(userIds, message):
  → 指定User IDリストに同一メッセージをpush送信
  → アクション履歴「絞り込み送信」
```

#### setup.gs

```
setupSpreadsheet():
  → 設定シート（ヘッダー行あり）・会員マスタ・アクション履歴を作成

setupNewEvent():
  → 4つのプロンプトで入力: 識別名・イベント名・開催日・募集終了日
  → {識別名}_応募シートの存在確認（先にフォーム連携が必要）
  → {識別名}_当落シートを自動作成
  → 設定シートに行を追加
```

## 3. スプレッドシート設計

### 3-1. シート構成

| シート名 | 数 | 用途 |
|---|---|---|
| 設定シート | 1（固定） | 全イベントの一覧・メッセージ文 |
| 会員マスタ | 1（固定・永続） | User ID・受付コード・名前の一元管理 |
| アクション履歴 | 1（固定・永続） | 全インタラクションのログ |
| {識別名}_応募 | イベント数分 | フォームの回答が自動記録される |
| {識別名}_当落 | イベント数分 | 当落入力・送信管理 |

### 3-2. 各シートの列定義

#### 設定シート

1行目はヘッダー。2行目以降にイベントを1行ずつ追加する。

| 列 | 内容 | 入力方法 |
|---|---|---|
| A | イベント名（表示用） | 手動 |
| B | 開催日（YYYY/MM/DD） | 手動 |
| C | 募集終了日（YYYY/MM/DD） | 手動 |
| D | 応募シート名（タブ名と完全一致） | setupNewEventが自動入力 |
| E | 当選メッセージ | 手動（任意。空欄時はデフォルト文） |
| F | 落選メッセージ | 手動（任意。空欄時はデフォルト文） |

#### 会員マスタ

旧称：LINE登録者シート。GASが「応募」受信時・フォーム送信処理時に自動書き込み・更新する。

| 列 | 内容 | 入力方法 |
|---|---|---|
| A | 登録日時 | 自動（「応募」Webhook受信時） |
| B | User ID | 自動（LINEから取得） |
| C | 受付コード（大文字英数字8文字） | 自動（GAS生成） |
| D | 備考 | 手動（任意） |
| E | 名前 | 自動（フォーム送信時にC列から取得して更新） |
| F | 最終更新日時 | 自動（フォーム送信時に更新） |

#### アクション履歴シート

| 列 | 内容 | 入力方法 |
|---|---|---|
| A | タイムスタンプ | 自動 |
| B | User ID | 自動 |
| C | アクション種別 | 自動 |
| D | イベント識別名 | 自動（該当なしの場合は空欄） |
| E | 詳細 | 自動 |

アクション種別の定義値：

| 値 | 発生タイミング |
|---|---|
| `受付コード発行` | 新規User IDから「応募」受信時 |
| `既存コード再送` | 既存User IDから「応募」受信時 |
| `応募状況照会` | 「応募状況」受信時 |
| `応募完了` | フォーム送信・突合成功時 |
| `突合失敗` | フォーム送信・突合失敗時 |
| `当落通知_当選` | 当選通知を送信した時 |
| `当落通知_落選` | 落選通知を送信した時 |
| `絞り込み送信` | ダッシュボードから絞り込み送信した時 |

#### {識別名}_応募シート

Googleフォームの回答が列A〜Rに自動記録される。GASはS・T列を追記する。

| 列 | 内容 | 入力方法 |
|---|---|---|
| A | タイムスタンプ | Google Forms自動 |
| B | 受付コード | フォーム回答（1問目） |
| C | お名前 | フォーム回答（2問目） |
| D〜R | その他フォーム項目 | フォーム回答（3〜17問目） |
| S | User ID | GAS自動追記（form-trigger.gs） |
| T | 応募完了通知済み | GAS自動追記「済」（form-trigger.gs） |

> **⚠️ フォーム設計の制約**
>
> GASは以下の列インデックスをハードコードしているため、フォームの設計変更時は注意が必要。
>
> | 参照箇所 | 依存する列 | 依存の理由 |
> |---|---|---|
> | 重複チェック（form-trigger.gs） | B列（index 1） | `appData[i][1]`で受付コードを読み取る |
> | ダッシュボード名前表示（dashboard.gs） | C列（index 2） | `appData[i][2]`でお名前を読み取る |
> | User ID参照・書き込み（全体） | S列（index 18、getRange列番号19） | 固定列番号で読み書きする |
> | 通知済みフラグ（form-trigger.gs） | T列（index 19、getRange列番号20） | 固定列番号で読み書きする |
>
> **フォームの質問数は17問まで（B〜R列）が上限。** 18問目以降を追加するとS・T列の位置がずれて動作しなくなる。
> **受付コードは必ず1問目、お名前は必ず2問目に配置すること。**
>
> ※ `e.namedValues['受付コード']` という名前参照は質問の順序に依存しないが、スプレッドシートの列読み取り（B・C列）は位置に依存する。

#### {識別名}_当落シート

GASがフォーム送信のたびにA・B列を転記。当落はダッシュボードまたは直接入力。送信後GASがD・E列を追記。

| 列 | 内容 | 入力方法 |
|---|---|---|
| A | お名前 | GAS自動転記（応募シートC列から） |
| B | User ID | GAS自動転記（応募シートS列から） |
| C | 結果（「当選」または「落選」） | ダッシュボードまたは手動 |
| D | 送信済み | GAS自動追記「済」（notify-result.gs） |
| E | 送信日時 | GAS自動追記（notify-result.gs） |

## 4. ダッシュボード設計

### 4-1. 実装方式

GAS HTMLサービス（HtmlService）を使用する。`dashboard.gs` の `doGet()` でHTMLを返す。

| 項目 | 設定 |
|---|---|
| デプロイ種別 | Webアプリ |
| 実行ユーザー | 自分 |
| アクセス権 | 組織内のユーザー（@southpaw.co.jpのGoogleアカウントを持つ全員がアクセス可能） |
| URLの共有 | ダッシュボードURLをスタッフ間で共有 |

### 4-2. 画面構成

```
ダッシュボード
├── イベント一覧タブ
│    ・設定シートの全イベントを表示
│    ・各イベント: イベント名・開催日・応募者数・当落済み数・未処理数
│    ・イベント名クリックで応募者一覧へ
│
├── 応募者一覧タブ（イベント選択後）
│    ・参加者一覧テーブル: 名前・応募日時・ステータス（未処理/当選/落選）・送信済み
│    ・各行に「当選」「落選」ボタン → クリックで当落シートのC列を更新
│    ・「当落通知を送信」ボタン → 未送信の全行に一括送信
│    ※表示対象：応募シートS列にUser IDが記録されている行のみ
│      受付コード突合に失敗した参加者はS列が空のため表示されない
│
└── 絞り込み送信タブ
     ・イベント選択
     ・ステータスでフィルター（未応募・応募済み・当選・落選）
       - 「応募済み」= 応募シートにUser IDあり かつ 当落結果が未入力の参加者のみ
       - 「当選」「落選」= 当落シートのC列に結果が入力済みの参加者
       - 「未応募」= 会員マスタには存在するが応募シートにUser IDがない参加者
     ・メッセージ入力エリア
     ・「送信」ボタン → 対象者全員にpush送信
```

### 4-3. クライアント・サーバー間の通信

GAS HTMLサービスでは `google.script.run` を使ってクライアントサイドJSからサーバー関数を呼び出す。

| サーバー関数 | 引数 | 戻り値 | 用途 |
|---|---|---|---|
| `getEventsData()` | なし | イベント一覧＋統計 | イベント一覧タブの表示 |
| `getApplicants(appSheetName, resultSheetName)` | シート名 | 応募者リスト | 応募者一覧の表示 |
| `setResult(resultSheetName, userId, result)` | シート名・User ID・結果 | 成功/失敗 | 当落の更新 |
| `sendResultsFromDashboard(resultSheetName)` | シート名 | 送信数 | 当落通知の一括送信 |
| `getFilteredUsers(appSheetName, resultSheetName, status)` | シート名・ステータス | User IDリスト | 絞り込み対象の取得 |
| `sendBroadcast(userIds, message)` | User IDリスト・メッセージ | 送信数 | 絞り込み送信の実行 |

## 5. LINE Messaging API設計

### 5-1. 使用するAPIエンドポイント

| 用途 | API種別 | 送信タイミング |
|---|---|---|
| 受付コード返信 | reply API | 「応募」受信時 |
| 応募状況返信 | reply API | 「応募状況」受信時 |
| 応募完了通知 | push API（個人宛） | フォーム送信時（突合成功時） |
| 当落結果通知 | push API（個人宛） | 送信ボタン押下時（ダッシュボードまたはメニュー） |

| スタッフログ通知 | push API（グループ宛） | 各処理完了時 |
| 受付コード不一致アラート | push API（グループ宛） | 突合失敗時 |

### 5-2. GmailAppの使用

| 用途 | 送信タイミング | 宛先 |
|---|---|---|
| 受付コード不一致アラートメール | 突合失敗時 | ALERT_EMAIL（スクリプトプロパティ） |

## 6. セキュリティ・認証設計

### 6-1. 認証情報の管理

| 項目 | プロパティキー | 説明 |
|---|---|---|
| チャネルアクセストークン | `LINE_CHANNEL_ACCESS_TOKEN` | LINE Messaging APIの認証トークン |
| チャネルシークレット | `LINE_CHANNEL_SECRET` | 将来的なWebhook署名検証用 |
| スタッフグループID | `STAFF_GROUP_ID` | LINEグループのGroup ID |
| スプレッドシートID | `SPREADSHEET_ID` | GoogleスプレッドシートのID |
| アラートメール宛先 | `ALERT_EMAIL` | 受付コード不一致アラートの送信先 |

### 6-2. ダッシュボードのアクセス制御

- GASのデプロイ設定で「組織内のユーザー」にすると @southpaw.co.jp のGoogleアカウントを持つ全員がアクセス可能
- この設定はGoogle Workspace（G Suite）を利用している組織でのみ選択肢に表示される
- 「全員」にすると誰でもURLさえ知っていればアクセスできるため注意が必要
- URLはスタッフ間で共有して運用する

## 7. エラーハンドリング

| エラーケース | 対処方法 |
|---|---|
| 受付コードがフォームと一致しない | スタッフLINEグループにアラート投稿 ＋ ALERT_EMAILにメール送信 |
| LINE送信失敗（API障害等） | エラーをLogger.logに記録。スタッフグループへの通知も失敗する可能性がある点を許容 |
| GAS実行時間超過（6分制限） | 50名以内の送信では問題なし。送信済みフラグにより途中再送が可能 |
| 同じ受付コードで2回フォーム送信 | 重複チェックでスタッフにアラート。2回目は処理しない |
| ダッシュボードの操作ミス | setResult・sendBroadcastは入力値を検証してから実行。結果は「当選」「落選」のみ許容 |

### スクリプトプロパティの必須・任意区分

| プロパティキー | 区分 | 未設定時の挙動 |
|---|---|---|
| `LINE_CHANNEL_ACCESS_TOKEN` | **必須** | LINE API呼び出しが全て認証エラーで停止 |
| `SPREADSHEET_ID` | **必須** | スプレッドシートへのアクセスが全て失敗・停止 |
| `STAFF_GROUP_ID` | 任意 | スタッフグループへのpush通知をスキップ（個人宛送信・アクション履歴は継続） |
| `ALERT_EMAIL` | 任意 | 突合失敗時のメール送信をスキップ（スタッフLINE通知は継続） |
| `LINE_CHANNEL_SECRET` | 任意（将来用） | 現時点では未使用 |

## 9. セットアップ手順（詳細）

### 9-1. GASプロジェクトの準備

1. Google Apps Script（script.google.com）で**新規プロジェクトを作成**
2. プロジェクト名を任意で設定（例：「テニスイベント予約管理」）
3. 以下の7ファイルをプロジェクト内に作成・貼り付ける（デフォルトの `コード.gs` は削除してよい）：
   - `config.gs` / `utils.gs` / `webhook.gs` / `form-trigger.gs` / `notify-result.gs` / `dashboard.gs` / `setup.gs`

### 9-2. スクリプトプロパティの設定

「プロジェクトの設定 → スクリプトプロパティ」で以下を設定：

| キー | 値 | 備考 |
|---|---|---|
| `LINE_CHANNEL_ACCESS_TOKEN` | LINEチャンネルのアクセストークン | LINE Developersコンソールから取得 |
| `LINE_CHANNEL_SECRET` | LINEチャンネルシークレット | LINE Developersコンソールから取得 |
| `STAFF_GROUP_ID` | スタッフLINEグループのGroup ID | ※取得方法は後述（9-7参照） |
| `SPREADSHEET_ID` | GoogleスプレッドシートのID | スプレッドシートのURLの `/d/` と `/edit` の間の文字列 |
| `ALERT_EMAIL` | `s.daito@southpaw.co.jp` | 受付コード不一致アラートの送信先 |

### 9-3. スプレッドシートのセットアップ

GASエディタで関数リストから `setupSpreadsheet` を選択して「実行」ボタンを押す。

**何が作成・更新されるか：**
- 「設定シート」が存在しない場合は新規作成（ヘッダー行を追加）
- 「LINE登録者シート」が存在する場合は「会員マスタ」に改名し、E列「名前」・F列「最終更新日時」のヘッダーを1行目に追加
- 「会員マスタ」が存在しない場合は新規作成
- 「アクション履歴」シートを新規作成（存在する場合はスキップ）

**既存の設定シートがある場合（追加作業）：**  
`setupSpreadsheet()` は既存の設定シートを変更しないため、手動で1行目に以下を追加：
- E1セル：`当選メッセージ`
- F1セル：`落選メッセージ`

### 9-4. GASのデプロイ設定

**重要：** webhook.gs と dashboard.gs は**同じGASプロジェクト内の別々のデプロイ**として設定する。  
GASは1プロジェクトに対して複数のデプロイ（異なるURLと設定を持つ）を作成できる。

**デプロイ①：LINE Webhook用（アクセス権：全員）**

「デプロイ → 新しいデプロイ → 種類：Webアプリ」で以下を設定：
- 実行ユーザー：自分
- アクセスできるユーザー：**全員**（LINEのサーバーがPOSTリクエストを送るために必須）
- 発行されたURLをコピーしてLINE Webhookに設定する（次のステップ）

このデプロイのURLで `doPost(e)` が実行される（「応募」「応募状況」の処理）。  
`doGet(e)` も同じURLで動作するが、LINE Webhook経由でのアクセスしか来ないため問題ない。

**デプロイ②：ダッシュボード用（アクセス権：自分のみ）**

再度「デプロイ → 新しいデプロイ → 種類：Webアプリ」で以下を設定：
- 実行ユーザー：自分
- アクセスできるユーザー：**自分のみ**（大藤さんのGoogleアカウントでのみアクセス可能）
- 発行されたURLをブラウザでブックマーク保存する

このデプロイのURLでブラウザからアクセスすると `doGet(e)` が実行される（ダッシュボードの表示）。

### 9-5. LINE WebhookのURL設定

LINE Developersコンソール（developers.line.biz）を開き：
1. 対象のチャンネル → 「Messaging API設定」タブ
2. 「Webhook URL」の欄に **デプロイ①のURL** を貼り付ける
3. 「検証」ボタンを押して「成功」が表示されることを確認
4. 「Webhookの利用」をONにする

### 9-6. 新イベントの追加手順

イベントごとに以下の手順を実施する：

**① Googleフォームの作成と連携**

1. Googleフォームを新規作成する
2. 質問を以下の順番で設定する（順番厳守：B列以降の列番号に直結する）：
   - 1問目：「受付コード」（短答式）
   - 2問目：「お名前」（短答式）
   - 3問目以降：その他の収集項目
3. フォームの「回答」タブ → 「スプレッドシートにリンク」をクリック
4. **既存のスプレッドシート**を選択し、新しいシートとして連携する
5. 自動作成されるシート名を **`{識別名}_応募`** に変更する（例：`コーチA6月_応募`）
   - ※この名前変更が必須。GASはこのシート名でイベントを識別する

**② セットアップの実行**

スプレッドシートのメニュー「イベント管理 → 新しいイベントをセットアップ」を実行し、ダイアログに以下を入力：
- 識別名（例：`コーチA6月`）← シート名のプレフィックス
- イベント名（例：`コーチAレッスン 6月15日`）← 参加者向け表示名
- 開催日（例：`2026/06/15`）
- 募集終了日（例：`2026/06/10`）

実行後、`{識別名}_当落` シートが自動作成され、設定シートにイベント行が追加される。

**③ フォームトリガーの設定（初回のみ）**

GASエディタの「トリガー（時計アイコン）」を開き、以下を設定：
- 実行する関数：`onFormSubmit`
- イベントのソース：**スプレッドシートから**（「フォームから」ではない）
- イベントの種類：フォーム送信時

※2つ目以降のイベントは既存のトリガーが自動で全フォームに対応するため、この作業は初回1回のみ。

### 9-7. STAFF_GROUP_ID の取得方法

スタッフLINEグループのGroup IDを取得するには：

1. スタッフグループにLINE公式アカウントを招待する（まだの場合）
2. グループ内で任意のメッセージを送信する
3. `webhook.gs` の `doPost(e)` の先頭に一時的に `Logger.log(JSON.stringify(e));` を追記してデプロイする（デプロイ①を「新バージョン」で更新）
4. グループのメッセージ受信後、GASのログ（「実行数」→「ログ」）を確認する
5. ログの `source.groupId` の値が Group ID
6. スクリプトプロパティ `STAFF_GROUP_ID` に設定する

> **⚠️ 取得後は必ず追記したコードを削除すること**
>
> `Logger.log(JSON.stringify(e))` はWebhookで受信する**全データ**（User ID・メッセージ内容を含む）をログに残す。  
> Group ID取得後は即座に当該行を削除し、再デプロイすること。削除前にスタッフグループ外からのメッセージを受信した場合、個人情報がログに記録される。

## 10. セキュリティに関する補足

### Webhook署名検証の省略について

LINE Messaging APIは、Webhookリクエストの正当性を確認するため `X-Line-Signature` ヘッダーを付与する。  
本システムでは以下の理由から署名検証を**省略**している：

- GAS（Google Apps Script）はHTTPリクエストのヘッダーに `UrlFetchApp` 経由以外でアクセスできない
- `doPost(e)` の引数 `e` にはヘッダー情報が含まれない（GASの仕様上の制限）
- GASのWebアプリURLは公開されているが、正しいJSON形式のイベントを送らないと `doPost` 内の処理でエラーになるため実質的な不正アクセスは困難

Phase2以降でセキュリティ要件が高まった場合は、署名検証に対応したサーバーランタイム（Cloud FunctionsやCloud Run等）への移行を検討する。
