                    ▼                    ▼

## 2. コンポーネント一覧

| コンポーネント | 種別 | 役割 |
|---|---|---|
| LINE公式アカウント | 外部サービス | 参加者との接点。「参加希望」「応募」「応募状況」の受信・通知送信 |
| LINE Messaging API | 外部API | push/reply送信・Webhook配信 |
| GmailApp | GAS組み込み | 受付コード不一致アラートのメール送信 |
| GAS Webアプリ（webhook.gs） | 実装 | 「応募」「応募状況」のWebhookを受信・処理 |
| GAS フォームトリガー（form-trigger.gs） | 実装 | フォーム送信時に自動実行。突合・通知・転記・会員マスタ更新 |
| GAS 当落送信（notify-result.gs） | 実装 | ダッシュボードまたはメニューから当落通知を一括送信 |
| GAS ダッシュボード（dashboard.gs） | 実装 | HTMLサービスで管理UIを配信。イベント一覧・当落入力・絞り込み送信 |
| GAS セットアップ（setup.gs） | 実装 | 初回セットアップ・新イベント追加 |
| Googleスプレッドシート | データストア | 設定シート・会員マスタ・アクション履歴・応募シート・当落シートを管理 |
| Googleフォーム | 外部ツール | 参加者情報の収集（イベントごとに1枚） |

## 3. データフロー詳細

### 3-1. 応募受付フロー（「応募」送信）

```
参加者
  │ 「応募」送信（前後空白トリム後に完全一致で判定）
  ▼
LINE Messaging API
  │ Webhookイベント配信
  ▼
webhook.gs（doPost関数）
  │ 会員マスタB列でUser IDを検索
  │
  ├─ 既存User IDあり（2回目の応募）：
  │   └─→ LINE reply API
  │        同行C列の受付コードをそのまま再送
  │        アクション履歴「既存コード再送」
  │
  └─ 新規：
      ├─→ 大文字英数字（A〜Z・0〜9、O/0/I/1を除く）8文字の受付コードを生成
      ├─→ 会員マスタ
      │    登録日時・User ID・受付コード・備考（空）・名前（空）・最終更新（空）
      ├─→ LINE reply API
      │    受付コードを参加者に返信
      └─→ アクション履歴「受付コード発行」
```

### 3-2. 応募状況照会フロー（「応募状況」送信）

```
参加者
  │ 「応募状況」送信（前後空白トリム後に完全一致で判定）
  ▼
webhook.gs（doPost関数）
  │ 設定シートの全イベント行（2行目以降）を走査
  │
  │ 各イベントについて：
  ├─ 開催日 < 今日：スキップ（表示しない）
  │
  └─ 開催日 >= 今日：ステータスを判定
      │
      │ {識別名}_当落シートB列でUser IDを検索
      ├─ C列が「当選」→ 応募済み（当選）
      ├─ C列が「落選」→ 応募済み（落選）
      │
      │ 当落シートになし → {識別名}_応募シートS列でUser IDを検索
      ├─ 存在する → 応募済み（当落発表前）
      │
      │ どこにも存在しない
      ├─ 募集終了日 >= 今日 → 未応募（応募期間中）
      └─ 募集終了日 < 今日 → 未応募（応募期間終了）

  全イベント分を1通のメッセージにまとめてreply送信
  アクション履歴「応募状況照会」
```

### 3-3. 応募完了通知フロー（フォーム送信）

```
参加者
  │ Googleフォームに入力・送信
  ▼
Googleフォーム送信イベント
  │ onFormSubmitトリガー発火
  ▼
form-trigger.gs
  │ B列（受付コード）を取得・大文字に変換
  │ e.range.getSheet().getName() でイベントの応募シート名を取得
  │ 重複チェック：既存行に同じコード＋T列「済」があればアラートして終了
  │ 会員マスタC列を走査（大文字変換して比較）
  │
  ├─ User IDが見つかった場合：
  │   ├─→ 応募シートS列にUser IDを記録
  │   ├─→ 応募シートT列に「済」を記録
  │   ├─→ {識別名}_当落シートA列にお名前・B列にUser IDを転記
  │   ├─→ 会員マスタE列（名前）・F列（最終更新日時）を更新
  │   ├─→ LINE push API（参加者個人宛）「応募完了」通知
  │   ├─→ LINE push API（スタッフグループ宛）応募受付完了ログ
  │   └─→ アクション履歴「応募完了」
  │
  └─ User IDが見つからない場合：
      ├─→ LINE push API（スタッフグループ宛）アラート
      ├─→ GmailApp → s.daito@southpaw.co.jp アラートメール
      └─→ アクション履歴「突合失敗」
```

### 3-4. 当落通知フロー

```
大藤さん
  │ ダッシュボードの応募者一覧で当選/落選をクリック
  │ または スプレッドシートの当落シートC列に直接入力
  │ 全員分を目視確認後「当落通知を送信」ボタン押下
  ▼
notify-result.gs
  │ 対象当落シートを2行目から走査
  │ D列が「済」の行はスキップ
  │ C列が「当選」「落選」以外の行はスキップ
  │ 設定シートE列・F列からイベントごとのメッセージ文を取得
  │
  ├─→ LINE push API（参加者個人宛）当選または落選メッセージ
  ├─→ 当落シートD列に「済」
  ├─→ 当落シートE列に送信日時を記録
  └─→ アクション履歴「当落通知_当選」または「当落通知_落選」

全員分の走査完了後：
  └─→ LINE push API（スタッフグループ宛）
       「当落通知送信完了 当選:N名 / 落選:N名」
```

### 3-5. ダッシュボード・絞り込み送信フロー

```
大藤さん
  │ ダッシュボードURLにアクセス
  ▼
dashboard.gs（doGet）
  │ GAS HTMLサービスでダッシュボードHTMLを配信

  【イベント一覧タブ】
  大藤さんがイベントを選択
    │ google.script.run → getEventsData()
    ▼
  設定シートを参照してイベント一覧・統計情報を返す
  画面に表示

  【応募者一覧タブ】
  大藤さんがイベントを選択
    │ google.script.run → getApplicants(appSheetName, resultSheetName)
    ▼
  応募シート・当落シートを参照して応募者リストを返す
  各行に当選/落選ボタンを表示

  大藤さんが当選/落選ボタンをクリック
    │ google.script.run → setResult(resultSheetName, userId, result)
    ▼
  当落シートC列を更新

  大藤さんが「当落通知を送信」ボタンをクリック
    │ google.script.run → sendResultsFromDashboard(resultSheetName)
    ▼
  notify-result.gsの処理を実行

  【絞り込み送信タブ】
  大藤さんがイベント・ステータスを選択してメッセージを入力
    │ google.script.run → getFilteredUsers() → sendBroadcast()
    ▼
  対象User IDリストにpush API送信
  アクション履歴「絞り込み送信」に記録
```

## 4. GASデプロイ構成

```
GASプロジェクト（1つ）
  ├── config.gs        ← 定数・スクリプトプロパティアクセサー
  ├── utils.gs         ← 共通ユーティリティ・LINE API・アクション履歴記録
  ├── webhook.gs       ← doPost(e)：LINE Webhookの受信・「応募」「応募状況」の処理
  ├── form-trigger.gs  ← onFormSubmit：フォーム送信時に自動実行
  ├── notify-result.gs ← sendResults：当落通知の一括送信・onOpen：メニュー追加
  ├── dashboard.gs     ← doGet(e)：ダッシュボードHTML配信・サーバー関数群
  └── setup.gs         ← setupSpreadsheet・setupNewEvent

【デプロイ構成の重要な考え方】

  同じGASプロジェクトから「デプロイ」を2つ作成する。
  GASはHTTPメソッドで処理を自動的に振り分ける：
    POST リクエスト → doPost(e) が実行される（LINEのWebhookが送ってくる）
    GET  リクエスト → doGet(e)  が実行される（ブラウザでダッシュボードを開く）

  デプロイ①：LINE Webhook用
    URL:          デプロイ後に発行されるWebアプリURL
    実行ユーザー: 自分
    アクセス権:   全員（LINEサーバーがPOSTを送るために必須）
    用途:         LINEのWebhook URLとして設定。POSTが来るとdoPost(e)を実行。

  デプロイ②：ダッシュボード用
    URL:          デプロイ後に発行されるWebアプリURL（デプロイ①とは別URL）
    実行ユーザー: 自分
    アクセス権:   組織内のユーザー（@southpaw.co.jpのGoogleアカウントを持つ全員）
    用途:         ブラウザで開くとdoGet(e)を実行してダッシュボードHTMLを返す。

スクリプトプロパティ（1プロジェクト共通）
  LINE_CHANNEL_ACCESS_TOKEN  ← LINE Messaging APIの認証トークン
  LINE_CHANNEL_SECRET        ← 将来的な署名検証用（現在は未使用）
  STAFF_GROUP_ID             ← スタッフLINEグループのGroup ID
  SPREADSHEET_ID             ← GoogleスプレッドシートのID
  ALERT_EMAIL                ← s.daito@southpaw.co.jp
```

## 5. スプレッドシート構成

```
スプレッドシート（1ファイル・シート数はイベント数に応じて増加）
  │
  ├── 設定シート（固定・1枚）
  │    1行目: ヘッダー
  │    2行目以降: イベントごとに1行ずつ追加
  │    A: イベント名 / B: 開催日 / C: 募集終了日 / D: 応募シート名 / E: 当選文 / F: 落選文
  │
  ├── 会員マスタ（固定・1枚・永続利用）
  │    ※旧称：LINE登録者シート
  │    A: 登録日時
  │    B: User ID
  │    C: 受付コード（大文字英数字8文字）
  │    D: 備考（手動・任意）
  │    E: 名前（フォーム送信時にGASが自動更新）
  │    F: 最終更新日時（フォーム送信時にGASが自動更新）
  │
  ├── アクション履歴（固定・1枚・永続利用）
  │    A: タイムスタンプ
  │    B: User ID
  │    C: アクション種別
  │    D: イベント識別名
  │    E: 詳細
  │
  ├── {識別名}_応募  ← Googleフォームの回答が自動記録（イベントごとに1枚）
  │    A: タイムスタンプ（Google Forms自動）
  │    B: 受付コード（1問目）← GASがUser ID突合に使用
  │    C: お名前（2問目）
  │    D〜R: その他フォーム項目（3〜17問目）
  │    S: User ID（GAS自動追記）
  │    T: 応募完了通知済み（GAS自動追記「済」）
  │
  └── {識別名}_当落（イベントごとに1枚）
       A: お名前（GAS自動転記）
       B: User ID（GAS自動転記）
       C: 結果（ダッシュボードまたは手動入力「当選」または「落選」）
       D: 送信済み（GAS自動追記「済」）
       E: 送信日時（GAS自動追記）

  ※当落シート名はGASが応募シート名から自動導出（_応募 → _当落）
  ※当落通知はダッシュボードまたはスプレッドシートのメニューから送信
```

## 6. Phase2 拡張イメージ

### Phase2-A：キャンセル自動受付

webhook.gsの「その他のテキスト」処理に「キャンセル」キーワードの分岐を追記するだけで対応可能。

```
参加者
  │ 「キャンセル」送信
  ▼
webhook.gs（doPost関数）
  │ 「キャンセル」を検出
  ├─→ 応募シート・当落シートのステータスを「キャンセル」に更新
  ├─→ LINE reply API（参加者宛）「キャンセルを受け付けました」
  ├─→ LINE push API（スタッフグループ宛）キャンセルログ
  └─→ アクション履歴「キャンセル」
```

### Phase2-B：LIFFアプリ

LINE内で開くWebアプリ（LIFF）を開発することで受付コード不要の応募フローを実現できる。
受付コードがなくてもLINEのUser IDを直接取得してフォームと紐付けることが可能。
