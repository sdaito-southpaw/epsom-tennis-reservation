## 2. 課題と目的

### 2-1. 現状の課題

| # | 課題 | 詳細 |
|---|---|---|
| 1 | 応募完了通知が手作業 | フォーム送信後、スタッフが個別にLINEで連絡しており工数がかかる |
| 2 | 当落連絡の取り違えリスク | 手動でLINEを送るため、送り間違いの可能性がある |
| 3 | 個人への確実な送信手段がない | LINE一斉送信はできるが、特定の個人に確実に届けるUser IDの管理ができていない |
| 4 | 送信履歴が残らない | 誰に何を送ったかの記録がなく、キャンセル対応時に確認できない |
| 5 | 応募者の情報が散在している | フォーム回答・LINE IDが別管理になっており横断的に確認できない |
| 6 | 当落入力・送信がスプレッドシート依存 | 複数シートを行き来する操作が複雑で、イベント数が増えるほど管理コストが上がる |

### 2-2. このシステムを作る目的

「参加者への連絡を確実に・個人宛に・履歴を残しながら半自動化することで、スタッフの手作業を削減し、取り違えゼロの運営体制を作る。さらに会員データを一元管理し、将来的なマーケティング活用の基盤を整える。」

## 5. 業務フロー

### 5-1. 参加者フロー

```
STEP 1: LINE公式アカウントを友だち追加して「参加希望」と送る
  → Googleフォームのリンクが自動返信で届く
  （LINE公式アカウントのキーワード返信機能・GAS不使用・設定済み）

STEP 1-5（任意）: 「応募状況」と送る
  → 現在の応募状況が自動返信で届く（応募済みか・当落結果が出ているか）

STEP 2: 「応募」と送る
  → 数秒後に受付コード（英数字8文字）が返信で届く
  → このコードを大切に保管する

STEP 3: 受付コードを手元に置いてGoogleフォームを開く
  → お名前・受付コード・その他必要事項を入力して送信する

STEP 4: フォーム送信後、LINEに「応募完了」通知が届く
  → このメッセージが届けば応募完了（参加者の操作はここで終わり）

STEP 5: 後日、LINEに当落通知が届く
  → 当選または落選のメッセージを受け取る
```

### 5-2. スタッフフロー

```
【イベント告知前】
  - 「イベント管理 > 新しいイベントをセットアップ」を実行
  - ダッシュボードにイベントが追加されることを確認

【応募受付期間中】
  - スタッフLINEグループを確認するだけ
  - フォーム送信があるたびにグループに通知が届く
  - 受付コード不一致のアラートが届いた場合は参加者に個別連絡

【応募締め切り後〜当落確定】
  ① ダッシュボードを開く
  ② 対象イベントを選択 → 応募者一覧が表示される
  ③ 各参加者の行で「当選」または「落選」をクリック
  ④ 全員分の入力・内容を目視確認
  ⑤「当落通知を送信」ボタンを1回押す
  ⑥ LINEグループに送信ログが流れることを確認

【随時】
  - ダッシュボードから絞り込み送信（例：未応募者への案内・当選者へのリマインダー）
```

### 5-3. 自動・半自動・手動の区分表

| 業務ステップ | 区分 | システムがすること | スタッフがすること |
|---|---|---|---|
| 「参加希望」へのフォームURL返信 | **自動** | LINEキーワード返信機能が返信（GAS不使用） | 操作不要 |
| 「応募状況」への自動返信 | **自動** | スプレッドシートを参照して現在の状況を返信 | 操作不要 |
| 受付コード発行 | **自動** | 「応募」受信 → コード生成・返信 | 操作不要 |
| User IDと会員情報の保存 | **自動** | 会員マスタに記録 | 操作不要 |
| 同一人物の2回目「応募」 | **自動** | 最初のコードを再送（上書きなし） | 操作不要 |
| 応募完了通知 | **自動** | 受付コード突合 → 個人宛にpush送信 | 操作不要 |
| 会員マスタへの名前自動更新 | **自動** | フォーム回答から名前を会員マスタに記録 | 操作不要 |
| アクション履歴への記録 | **自動** | 全インタラクションを自動ログ | 操作不要 |
| スタッフへのログ通知 | **自動** | LINEグループへ自動投稿 | グループで確認するだけ |
| 当落管理シートへの転記 | **自動** | フォーム送信のたびリアルタイムで転記 | 操作不要 |
| 新イベントの設定追加 | **半自動** | 当落シート自動作成・設定シートへの行追加 | メニューから4項目入力するだけ |
| 当落の決定 | **手動** | なし | ダッシュボードから当選/落選を選択 |
| 送信前レビュー | **手動** | なし | 全員分を目視確認 |
| 当落通知送信 | **半自動** | ボタン押下で全員に一括送信 | 「送信」ボタンを1回押す |
| 絞り込み送信 | **半自動** | 条件でフィルタリングして送信 | ダッシュボードで条件選択・文面入力・送信 |
| 受付コード不一致の対応 | **手動** | スタッフLINEグループへアラート＋メール送信 | 参加者に個別連絡 |

## 6. 各ステップの業務詳細

### 6-1. STEP1：受付コード発行（「応募」送信）

| 項目 | 内容 |
|---|---|
| トリガー | 参加者がLINE公式アカウントに「応募」と送信 |
| 「応募」のマッチング条件 | 前後の空白を取り除いたうえで「応募」と完全一致した場合のみ反応する |
| 「応募状況」のマッチング条件 | 前後の空白を取り除いたうえで「応募状況」と完全一致した場合のみ反応する |
| 処理 | ランダム大文字英数字8文字の受付コードを生成してreply送信、User IDとコードを会員マスタに記録。使用文字はA〜Z・2〜9の32文字（紛らわしいO・0・I・1を除外） |
| フォームでの受付コード入力 | 大文字・小文字どちらで入力されても突合できるようにする（大文字に統一して比較） |
| 同じ人が2回「応募」を送った場合 | 最初のコードを再送する（コードは上書きしない・1アカウント1コード） |
| 「応募」以外のメッセージが届いた場合 | 無視する（返信しない）※「参加希望」はLINE公式の別機能で処理 |
| 受付コードの有効期限 | なし（いつでも使用可能） |

### 6-1-2. 「応募状況」送信時の返信ロジック

設定シートに登録された**全イベント**を対象に、1通のメッセージでまとめて返信する。

**表示対象の絞り込み：**

| 条件 | 表示 |
|---|---|
| 開催日 < 今日 | 表示しない（終了済みイベント） |
| 開催日 >= 今日 | 以下のステータスで表示 |

**ステータス判定（表示対象イベントごとに判定）：**

| 確認順 | 条件 | ステータス表示 |
|---|---|---|
| 1 | 当落管理シートにUser IDあり・C列が「当選」 | 応募済み（当選） |
| 2 | 当落管理シートにUser IDあり・C列が「落選」 | 応募済み（落選） |
| 3 | 応募一覧シートのS列にUser IDあり（当落未発表） | 応募済み（当落発表前） |
| 4 | どこにも存在しない・募集終了日 >= 今日 | 未応募（応募期間中） |
| 5 | どこにも存在しない・募集終了日 < 今日 | 未応募（応募期間終了） |

### 6-2. STEP2：Googleフォーム応募

| 項目 | 内容 |
|---|---|
| 参加者の操作 | Googleフォームの「受付コード」欄に、LINEで受け取ったコードをそのまま入力して送信 |
| 受付コードを紛失した場合 | LINE公式アカウントに再度「応募」と送ると同じコードが届く旨をフォーム上に案内する |
| フォームの構成 | イベントごとに1つ作成。回答先をそのイベントの応募シートに連携する |

#### フォーム収集項目一覧

| フォーム順（列） | 項目名 | 備考 |
|---|---|---|
| A（自動付与） | タイムスタンプ | Google Forms が自動記録 |
| B（1問目） | **受付コード** | システムがUser ID突合に使用・先頭に配置 |
| C（2問目） | お名前 | 会員マスタにも自動反映 |
| D（3問目） | ふりがな | |
| E（4問目） | 年齢 | |
| F（5問目） | 性別 | |
| G（6問目） | メールアドレス | |
| H（7問目） | お電話番号 | |
| I（8問目） | 緊急連絡先 | |
| J（9問目） | 応募イベントのコーチのレッスンについて | |
| K（10問目） | イベントを知ったきっかけ | |
| L（11問目） | 参加応募のきっかけ | |
| M（12問目） | テニスレベル | |
| N（13問目） | テニスをプレイする頻度 | |
| O（14問目） | テニス歴 | |
| P（15問目） | 普段テニスをしている地域 | |
| Q（16問目） | テニスをプレイしている環境 | |
| R（17問目） | 利用規約／プライバシーポリシーの確認 | |
| S（GAS追記） | User ID | フォーム送信後にGASが自動追記 |
| T（GAS追記） | 応募完了通知済み | 通知送信後にGASが「済」を自動追記 |

> **⚠️ フォーム設計の制約（順序・問数）**
>
> - **受付コードは必ず1問目に配置すること。** GASは応募シートのB列（1問目のフォーム回答が記録される列）を受付コードとして読み取る。別の列にあると突合が失敗する。
> - **お名前は必ず2問目に配置すること。** GASはC列をお名前として管理ダッシュボードに表示する。
> - **フォームの質問は17問まで（D列〜R列）が上限。** GASはS列（19列目）にUser ID、T列（20列目）に通知済みフラグを自動追記する。18問目以降を追加すると列番号がずれてシステムが動作しなくなる。フォームの質問を追加・削除する際は必ず開発担当者に確認すること。

### 6-3. STEP3：応募完了通知

| 項目 | 内容 |
|---|---|
| タイミング | フォーム送信と同時（ほぼリアルタイム） |
| 処理 | B列（受付コード）でUser IDを突合 → 個人宛にpush送信 → スタッフグループにログ投稿 → 会員マスタに名前を記録 |

### 6-4. STEP4：当落確定・送信

| 項目 | 内容 |
|---|---|
| 当落の決定者 | 大藤さん（スタッフ判断・システムは自動化しない） |
| 入力場所 | ダッシュボードの応募者一覧から当選/落選をクリック（スプレッドシート直接入力も可） |
| 使用する状態値 | 「当選」「落選」の2種類のみ（Phase1では「補欠」「キャンセル」は対象外） |
| 送信前のチェック | 全行に結果が入力されているか・入力ミスがないかを目視確認してからボタンを押す |
| 当落送信ボタンを2回押した場合 | 送信済みフラグが空の行のみ送信する。「済」が入っている行はスキップするため途中再送にも使える |
| メッセージ文面 | 設定シートE列（当選文）・F列（落選文）に記載したものを使用。空欄の場合はデフォルト文を使用 |

### 6-5. ダッシュボード機能

| 機能 | 内容 |
|---|---|
| イベント一覧 | 設定シートのイベントを一覧表示。応募者数・当落済み数・未処理数を確認できる |
| 応募者一覧 | イベントを選択すると応募者リストが表示される。名前・ステータス・応募日時を確認できる |
| 当落入力 | 応募者リストの各行から当選/落選をクリックで選択できる |
| 当落通知送信 | 未送信の当落を一括でLINE送信する |
| 絞り込み送信 | ステータス（未応募・応募済み・当選・落選）で対象者を絞り込んでメッセージを送信する。「応募済み」は当落結果がまだ入力されていない参加者のみを指す（当選・落選は別ステータス） |

## 7. 例外ケースと対応方針

| ケース | 検知方法 | スタッフの対応アクション |
|---|---|---|
| 受付コード不一致（フォームへの誤入力） | スタッフLINEグループへアラート投稿＋s.daito@southpaw.co.jpへメール送信 | 参加者に個別連絡し正しいコードを確認。再フォーム送信を依頼 |
| 参加者がLINE公式アカウントを友だち追加していない | 「コードが届かない」と問い合わせ | LINE公式アカウントのQRを案内し、友だち追加後に「応募」を送るよう依頼 |
| 重複フォーム送信（同じ受付コードで2回送信） | スタッフLINEグループへアラート投稿 | 最初の送信を有効として、2回目は処理しない |
| 当落通知の送信が途中で止まった | 送信ログが途中で止まっている | ダッシュボードから再度「送信」ボタンを押す（送信済みフラグで重複送信を防ぐ） |
| キャンセル連絡が来た | 参加者からLINEまたは直接連絡 | Phase1では手動対応。アクション履歴の備考欄に記録する |
| 受付コード突合に失敗した参加者のダッシュボード表示 | （検知できない）| 突合失敗者はGASが当落シートに書き込まないため、管理ダッシュボードの応募者一覧に表示されない。スタッフLINEグループのアラートから手動で確認・対応する |

## 8. LINEメッセージ文面定義

### 受付コード返信

```
ご応募ありがとうございます！
あなたの受付コードは

【 XXXXXXXX 】

です。
Googleフォームの「受付コード」欄にこのコードを入力してください。
```

### 応募完了通知

```
応募を受け付けました！

当落結果は後日このLINEでお知らせします。
しばらくお待ちください。
```

### 当選通知・落選通知

設定シートE列（当選文）・F列（落選文）に記載した文面をそのまま送信する。
空欄の場合は以下のデフォルト文を使用する。

**デフォルト当選文:**
```
【当選のお知らせ】
このたびはイベントへの参加が確定しました！

```

## 9. スプレッドシートを使う業務上の理由

| 理由 | 詳細 |
|---|---|
| 当落の決定はスタッフが行う | 誰を当選・落選にするかはスタッフが判断する必要があり完全自動化しない |
| 送信前に全員分を一覧で確認できる | ボタンを押す前に全員分の当落を目視確認できるため取り違えをゼロにできる |
| 送信履歴が証跡として残る | 送信済みフラグ・送信日時が記録されるため「誰に何を送ったか」を確認できる |
| データを直接確認・修正できる | ダッシュボード経由だけでなくスプレッドシートからも直接操作できる |
| 将来的な分析・活用が可能 | 会員マスタ・アクション履歴が蓄積されることでマーケティング活用の基盤になる |

## 10. スプレッドシート構成

スプレッドシートは1ファイル。シート数はイベント数に応じて増える。

### シート一覧

| シート名 | 数 | 説明 |
|---|---|---|
| 設定シート | 1（固定） | 全イベントの一覧を管理 |
| 会員マスタ | 1（固定・永続） | 全参加者のUser ID・受付コード・名前を管理。イベントをまたいで使い回す |
| アクション履歴 | 1（固定・永続） | 全インタラクションのログ |
| {イベント識別名}_応募 | イベント数分 | Googleフォームの回答が自動記録される |
| {イベント識別名}_当落 | イベント数分 | 当落入力・通知送信管理 |

例：`コーチA6月_応募`、`コーチA6月_当落`

### 設定シート

1行目はヘッダー行。2行目以降にイベントを1行ずつ追加する。終了したイベントの行は削除しない。

| 列 | 項目 | 記入者 | 例 |
|---|---|---|---|
| A | イベント名 | 大藤さん | コーチA 6月15日テニスイベント |
| B | 開催日 | 大藤さん | 2026/06/15 |
| C | 募集終了日 | 大藤さん | 2026/06/10 |
| D | 応募シート名 | 自動入力（setupNewEvent） | コーチA6月_応募 |
| E | 当選メッセージ | 大藤さん（任意） | 【当選】集合は10:00に... |
| F | 落選メッセージ | 大藤さん（任意） | 【落選】今回は... |

※当落シート名はGASが応募シート名から自動導出する（`_応募` → `_当落`）

### 会員マスタ

GASが「応募」受信時・フォーム送信処理時に自動書き込み・更新する。

| 列 | 項目 | 記入者 |
|---|---|---|
| A | 登録日時 | GAS自動 |
| B | User ID | GAS自動 |
| C | 受付コード | GAS自動 |
| D | 備考 | 手動（任意） |
| E | 名前 | GAS自動（フォーム送信時に更新） |
| F | 最終更新日時 | GAS自動 |

### アクション履歴シート

| 列 | 項目 | 記入者 |
|---|---|---|
| A | タイムスタンプ | GAS自動 |
| B | User ID | GAS自動 |
| C | アクション種別 | GAS自動 |
| D | イベント識別名 | GAS自動 |
| E | 詳細 | GAS自動 |

アクション種別の値：`受付コード発行` / `既存コード再送` / `応募状況照会` / `応募完了` / `突合失敗` / `当落通知_当選` / `当落通知_落選` / `絞り込み送信`

### {イベント識別名}_応募シート

Googleフォームの回答が自動記録される。GASはS・T列を追記する。

| 列 | 項目 | 記入者 |
|---|---|---|
| A | タイムスタンプ | Google Forms自動 |
| B | 受付コード（1問目） | フォーム回答 |
| C | お名前（2問目） | フォーム回答 |
| D〜R | その他フォーム項目（3〜17問目） | フォーム回答 |
| S | User ID | GAS自動追記 |
| T | 応募完了通知済み | GAS自動追記（「済」） |

### {イベント識別名}_当落シート

GASがフォーム送信のたびA・B列を転記。当落はダッシュボードまたは直接入力。送信後にGASがD・E列を追記。

| 列 | 項目 | 記入者 |
|---|---|---|
| A | お名前 | GAS自動転記 |
| B | User ID | GAS自動転記 |
| C | 結果 | ダッシュボードから入力（「当選」または「落選」）またはスプレッドシートに直接入力 |
| D | 送信済み | GAS自動追記（「済」） |
| E | 送信日時 | GAS自動追記 |

## 14. 用語定義

| 用語 | 定義 |
|---|---|
| 受付コード | 参加者を一意に識別するためのランダム英数字8文字コード。「応募」送信時に自動発行される |
| User ID | LINEが各ユーザーに割り当てる内部ID。push送信に必要。表示名とは異なる |
| push送信 | LINE Messaging APIを使い、特定のUser IDに対して送信するメッセージ |
| Webhook | LINEからのメッセージをGASが受信するためのHTTPエンドポイント |
| フォームトリガー | Googleフォームに回答が送信されたときにGASが自動実行される仕組み |
| 送信ログ | 誰にいつ何を送ったかの記録。アクション履歴・スタッフLINEグループの両方に残る |
| 設定シート | 全イベントの一覧（イベント名・開催日・募集終了日・応募シート名・当選文・落選文）を1行ずつ管理するシート |
| 募集終了日 | 参加者がフォームで応募できる最終日。開催日とは別に設定する |
| イベント識別名 | シート名のプレフィックス。例: `コーチA6月`。応募シートは`コーチA6月_応募`、当落シートは`コーチA6月_当落` |
| 会員マスタ | User ID・受付コード・名前を一元管理するシート。旧称：LINE登録者シート |
| アクション履歴 | 参加者の全インタラクションを時系列で記録するシート |
| ダッシュボード | GAS HTMLサービスで作成する管理Web画面。イベント管理・当落入力・送信・絞り込みが可能 |
